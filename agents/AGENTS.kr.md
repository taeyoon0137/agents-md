# 외부 AGENTS.md 생성 지침

이 파일은 `agents/AGENTS.kr.md`에 저장된 한국어 외부 참조 지침입니다.

다른 저장소나 문서 묶음의 `AGENTS.md`를 만들거나 갱신할 때만 사용합니다. 이 파일은 `agents-md` 저장소 자체의 유지보수 지침이 아닙니다. `agents-md` 자체를 수정할 때는 루트 `AGENTS.md`를 사용합니다.

## 레퍼런스 사용 규칙

사용자나 다른 지시가 이 문서를 레퍼런스하라고 했다는 것은, 이 문서의 지침을 따르라는 뜻입니다. 이 문서의 구조, 섹션 순서, 예시, 파일 배치, `agents-md` 저장소 고유 규칙을 대상 `AGENTS.md`에 그대로 복사하라는 뜻이 아닙니다.

대상 `AGENTS.md`는 확인된 대상 맥락에 맞게 작성합니다. 먼저 대상을 조사하고, 작업 실패를 막는 데 필요한 최소한의 지침을 작성합니다.

## 대상 범위

대상은 아래 중 하나일 수 있습니다.

- 코드 저장소
- 문서 중심 저장소
- 운영 문서 묶음
- 에이전트 지침이 필요한 기타 프로젝트

확인하지 않은 기술 스택, 명령어, 배포 방식, 조직명, 채널, URL, 파일 구조를 임의로 쓰지 않습니다.

## 언어 라우팅과 대상 문서 언어

- 대상 지침의 기본 언어는 영어입니다.
- 대상 언어를 확인할 수 없으면 영어로 작성합니다.
- 사용자의 선호 언어, 작업 환경, 주요 의사소통 환경이 한국어라면 `agents/AGENTS.kr.md`를 기준으로 하고 대상 지침을 한국어로 작성합니다.
- 대상 언어는 사용자 메시지, 기존 README/docs/contributing 문서, 기존 `AGENTS.md` 또는 `CLAUDE.md`, 커밋/이슈/PR 문체처럼 대상 안에서 확인되는 근거로만 추론합니다.
- 에이전트는 기본적으로 사용자의 선호 언어를 추론해 그 언어로 응답합니다.
- 사용자가 응답 언어나 대상 문서 언어를 명시하면 그 지시를 우선합니다.
- 이미 존재하는 대상 `AGENTS.md`의 언어와 추론한 대상 언어가 다르면, 사용자가 전체 번역이나 언어 전환을 명시하지 않는 한 기존 문서 언어를 보존하고 필요한 변경만 최소로 적용합니다.

새 대상에 언어 라우팅 구조를 만들고, 기존 대상 관례와 충돌하지 않는다면 아래처럼 작성합니다.

- 대상 루트 `AGENTS.md`는 영어로 작성합니다.
- 대상 루트 `AGENTS.md`는 라우터와 에이전트 역할 설명만 담당합니다.
- 세부 대상 지침은 `agents/AGENTS.en.md`, `agents/AGENTS.kr.md`처럼 `agents/AGENTS.<language>.md`에 둡니다.

기존 대상 지침을 갱신할 때는 아래를 따릅니다.

- 사용자가 언어 라우팅 도입이나 전체 재작성을 요청하지 않았다면 기존 구조를 보존합니다.
- 이 참조 파일이 이런 구조를 갖고 있다는 이유만으로 대상 구조를 바꾸지 않습니다.

## 먼저 읽기

이 파일은 외부 참조 상황, 즉 다른 저장소나 문서 묶음의 대상 `AGENTS.md`를 만들거나 갱신할 때만 사용합니다.

- [공통 원칙](#공통-원칙)과 [외부 참조 지침](#외부-참조-지침)을 읽고 적용합니다.
- 외부 참조 시 `agents-md` 저장소의 내부 유지보수 규칙을 대상에 복사하지 않습니다.
- 사용자가 별도로 다른 지시를 내리면 최신 사용자 지시를 우선합니다.
- 대상 안에 기존 `AGENTS.md`, 하위 디렉터리 지침, `CLAUDE.md`, README, contributing 문서가 있다면 함께 확인하고 충돌 여부를 보고합니다.

## 공통 원칙

아래 원칙은 대상 `AGENTS.md`에 반영할 수 있는 기본 원칙입니다. 대상에 맞는 내용만 골라 실제 맥락에 맞게 작성합니다.

### 응답 원칙

- 기본 응답 언어는 사용자의 선호 언어를 추론해 정합니다.
- 결론을 먼저 말하고, 확인하지 못한 내용은 명확히 분리합니다.
- 불필요한 칭찬, 감탄, 장식적 표현은 사용하지 않습니다.
- 검증하지 않은 내용을 확신하듯 말하지 않습니다.
- 명령어, 테스트, 배포, 파일 상태는 실제로 실행하거나 확인한 경우에만 성공했다고 보고합니다.

### 작업 시작 원칙

- 답변이나 수정을 시작하기 전에 관련 파일과 기존 문서를 먼저 읽습니다.
- 저장소 구조, API, 환경 변수, 명령어, 문서 생성 방식, 배포 절차를 추측하지 않습니다.
- 작업이 단순하지 않다면 먼저 짧은 계획을 세웁니다. 계획에는 변경 가능성이 있는 파일, 기대 동작, 검증 방법을 포함합니다.
- 요청이 모호하면 안전한 최소 해석을 선택하고, 선택한 해석을 명시합니다.
- 사용자가 명시적으로 요청하지 않은 큰 구조 변경이나 광범위한 리팩터링을 하지 않습니다.

### 변경 원칙

- 가능한 한 작고 국소적인 변경을 만듭니다.
- 기존 동작, 공개 API, 문서의 source of truth를 보존합니다.
- 파일명 변경, 디렉터리 이동, 새 추상화, 새 의존성 추가는 필요한 경우에만 합니다.
- 요청과 직접 관련 없는 포맷 변경을 하지 않습니다.
- 의존성이 실제로 바뀌지 않았다면 lockfile을 수정하지 않습니다.
- 생성 파일은 대상의 관례가 요구할 때만 직접 수정하고, 가능한 경우 원본 파일을 수정한 뒤 생성 명령을 실행합니다.
- 이미 수정된 파일은 사용자 또는 이전 작업의 변경으로 간주하고 임의로 되돌리지 않습니다.

### 보안과 기록 원칙

- secret, token, credential을 저장소에 기록하지 않습니다.
- production config, 배포 설정, analytics key, 운영 접근 정보는 사용자의 명시적 요청과 대상 문서의 근거가 없으면 수정하지 않습니다.
- 사용자가 명시적으로 요청하지 않은 내용을 커밋 메시지, 코드 주석, PR/이슈 본문, 문서, 설정 파일에 임의로 삽입하지 않습니다. 특히 다음은 사용자의 명시적 지시가 없으면 추가하지 않습니다.
  - `Co-Authored-By`, `Signed-off-by` 등 trailer
  - 에이전트, 모델, 제공사 이름이나 이메일
  - "Generated with ...", "Made by AI" 등 생성 도구 표기
  - 작업과 무관한 광고, 홍보, 외부 링크
- 위 항목을 넣어야 할 합당한 이유가 있다고 판단되더라도 먼저 사용자에게 묻고 승인받은 뒤에만 추가합니다.

### 검증 원칙

- 변경 후 대상에 존재하는 가장 관련성 높은 검증을 실행합니다.
- 일반적인 우선순위는 type check, lint, unit test, integration/e2e test, 문서 생성 검증입니다.
- 대상에서 실제 명령어를 확인하지 못했다면 명령어를 임의로 만들지 않습니다.
- 명령어가 없거나 환경 문제로 실행할 수 없다면 이유를 보고합니다.
- 실패한 검증을 무시하지 않습니다. 실패 원인과 변경 관련 여부를 분리해 설명합니다.

### 최종 응답 형식

작업을 마친 뒤 항상 아래 항목을 포함합니다.

1. 변경 요약
2. 변경 파일
3. 실행한 검증 명령과 결과
4. 남은 위험, 한계, 후속 필요 사항

코드 변경이 없거나 문서만 변경했다면 그 사실을 명시합니다.

## 외부 참조 지침

이 파일을 그대로 복사하지 않습니다. 대상 저장소나 문서 묶음을 조사한 뒤 대상의 실제 맥락에 맞는 `AGENTS.md`를 작성합니다.

### 대상 확인

대상에서 확인 가능한 내용만 조사합니다.

- 대상이 Git 저장소라면 `git status --short`
- 실제 파일 구조
- README, docs, contributing 문서
- 언어별 패키지 매니페스트와 lockfile이 있다면 해당 파일
- 설정 파일, CI 파일, script 디렉터리
- 사용 언어, 프레임워크, 문서 도구
- 빌드, 테스트, 린트, 포맷, 문서 생성 명령어
- 배포, 릴리스, 운영 반영 방식이 문서화되어 있는지
- 생성 파일과 source of truth의 관계
- 기존 `AGENTS.md`, 하위 디렉터리 지침, `CLAUDE.md`
- 반드시 참조해야 할 별도 문서의 위치와 우선순위
- 사용자의 작업 환경이나 주요 의사소통 언어를 확인할 수 있는 문서나 맥락
- 사용자가 Codex 품질, 모델, reasoning 설정 개선도 요청한 경우에만 `.codex/config.toml` 또는 관련 설정 파일

확인하지 못한 항목은 대상 `AGENTS.md`에 사실처럼 쓰지 않습니다. 필요하면 최종 응답에서 미확인 항목으로 분리합니다.

### 대상 AGENTS.md 작성 원칙

- 대상의 실제 역할, 파일 구조, 명령어, 검증 기준을 구체적으로 씁니다.
- 대상 문서 언어는 사용자의 작업 환경이나 주요 의사소통 언어에 맞춥니다.
- 대상 언어를 추론할 수 없으면 영어로 작성합니다.
- 이 언어별 지침 파일을 그대로 복사하지 말고, 대상에 해당하지 않는 항목은 제거합니다.
- 생성된 대상 `AGENTS.md`에는 `agents-md` 저장소를 계속 참고하라는 문구나 링크를 남기지 않습니다.
- 확인된 기존 지침이 있다면 우선순위를 명확히 쓰고, 충돌하는 새 규칙을 임의로 추가하지 않습니다.
- "친절하게 답하라" 같은 추상적 스타일보다 작업 실패를 막는 규칙을 우선합니다.
- 실제로 확인한 명령어만 포함합니다.
- 기술별 규칙은 대상에 해당할 때만 포함합니다.
- 문서 중심 대상은 코드 빌드/테스트 규칙보다 원본 문서, 생성 결과물, 리뷰 절차, 링크 확인, 게시/배포 절차를 중심으로 작성합니다.

### 기존 대상 AGENTS.md 갱신 원칙

대상에 이미 `AGENTS.md`가 있고 사용자가 이 문서를 기준으로 갱신하라고 요청하면, 신규 작성이 아니라 기존 지침의 보존적 업데이트로 처리합니다.

- 기존 `AGENTS.md`, 하위 `AGENTS.md`, `CLAUDE.md`, README, docs, contributing 문서를 먼저 읽습니다.
- 기존 문서에서 확인된 대상 고유 역할, 명령어, 경로, 검증 절차, 금지사항을 보존합니다.
- 이 문서는 적용 가능한 규칙과 실패 방지 원칙의 출처로 사용하되, 복사할 구조로 사용하지 않습니다.
- 대상 고유 사실을 일반 문구로 덮어쓰지 않습니다.
- 기존 지침과 이 문서가 충돌하면 충돌을 보고하고, 사용자의 명시적 지시 없이 대상 고유 규칙을 제거하지 않습니다.
- 의미를 보존할 수 있는 중복 규칙은 병합합니다.
- 오래되었거나 검증되지 않은 명령어를 조용히 사실처럼 고쳐 쓰지 말고, 확인 필요 항목으로 분리합니다.
- 사용자가 전체 재작성이나 언어 전환을 요청하지 않았다면 기존 문서 언어와 구조를 가능한 한 보존합니다.
- 사용자가 요청했거나 대상이 이미 그런 구조를 사용할 때만 루트/언어별 파일 라우팅을 도입합니다.
- 갱신 후 가능하면 `git diff --check`, `git status --short`, 대상에서 확인된 검증 명령을 실행합니다.
- 최종 응답에서는 보존한 기존 규칙, 새로 적용한 규칙, 확인하지 못한 항목, 검증 결과를 분리합니다.

- 디자인 시스템 또는 UI 기준

- 검증 기준

### 대상에 반영할 실패 방지 규칙

아래 규칙은 대상에 맞게 변환해 반영합니다.

- 수정 전에 관련 파일을 먼저 읽습니다.
- 구현 전에 기존 패턴을 확인합니다.
- 작은 diff를 우선합니다.
- 사용자가 요청하지 않은 큰 리팩터링을 하지 않습니다.
- 새 구성요소, 유틸리티, 타입, 문서 구조를 만들기 전에 기존 항목을 검색합니다.
- source-of-truth 파일과 생성 파일의 관계를 보존합니다.
- 테스트나 검증을 실행하지 않았다면 통과했다고 말하지 않습니다.
- type check, lint, test, 문서 생성 검증 실패를 무시하지 않습니다.
- 불확실성을 숨기지 말고 무엇을 확인했는지 밝힙니다.

### 커밋 원칙

대상 `AGENTS.md`에 커밋 규칙을 넣기 전에 대상 문서와 기존 지침에 이미 커밋 규칙이 있는지 확인합니다.

- 대상에 커밋 규칙이 있으면 그 규칙을 우선합니다.
- 대상 규칙과 이 문서의 기본 규칙이 충돌하면 임의로 병합하지 말고 사용자에게 묻습니다.
- 대상에 별도 규칙이 없고 사용자가 커밋 규칙 포함을 원하면 아래 기본값을 사용할 수 있습니다.

기본 커밋 규칙은 다음과 같습니다.

- 사용자가 명시적으로 커밋을 요청할 때만 커밋합니다.
- `Feat`, `Docs`, `Refactor`, `Fix`, `Chore`, `Test` prefix와 영문 Sentence Case 커밋 메시지를 사용합니다.
- 한 커밋에는 하나의 목적만 담습니다.
- 관련 없는 변경은 stage하지 않습니다.
- 같은 파일에 사용자 변경과 작업 변경이 섞여 있으면 필요한 hunk만 stage합니다.

### 큰 작업 계획

대상이 복잡한 변경을 자주 다루거나 사용자가 계획 파일을 원할 때만 대상 `AGENTS.md`에 `PLANS.md` 규칙을 포함합니다. 대상 관례가 확인되지 않았다면 파일 생성을 필수로 만들지 말고, 큰 작업은 먼저 계획해야 한다고만 씁니다.

`PLANS.md`를 포함할 때는 아래 항목을 후보로 사용합니다.

- 문제 정의
- 현재 동작
- 기대 동작
- 확인할 파일
- 구현 단계
- 검증 계획
- 롤백 위험

세 파일을 넘게 수정하거나, 아키텍처나 데이터 흐름을 바꾸거나, 운영 절차에 영향을 주거나, 디버깅이 필요한 작업은 구현 전에 계획을 명확히 해야 한다고 씁니다.

### 디자인 시스템 또는 UI 대상

UI, 프론트엔드, 디자인 시스템이 대상에서 확인될 때만 이 섹션을 포함합니다.

- 새 컴포넌트를 만들기 전에 유사한 기존 컴포넌트를 찾습니다.
- 디자인 시스템 primitive, 공유 컴포넌트, token을 우선 사용합니다.
- 기존 관례가 아니라면 color, font size, spacing, z-index, radius, shadow를 하드코딩하지 않습니다.
- 작업에서 명시적으로 요구하지 않으면 시각적 위계를 바꾸지 않습니다.
- 반응형 동작과 접근성을 보존합니다.
- 관련 있을 때 loading, empty, error, disabled, long-text 상태를 확인합니다.
- UX copy는 제품의 기존 용어와 톤에 맞춥니다.

### API 또는 백엔드 대상

API, 서버, datastore, 백엔드 작업이 대상에서 확인될 때만 이 섹션을 포함합니다.

- 명시적으로 요청받지 않았다면 backward compatibility를 보존합니다.
- 경계에서 입력을 검증합니다.
- 오류를 조용히 삼키지 않습니다.
- 사용자가 명시적으로 요청했거나 변경에 필요한 경우에만 database schema 또는 migration을 수정합니다.
- 별도 근거 없이 운영 설정이나 secret을 수정하지 않습니다.

### 버그 수정 기준

대상이 코드나 동작 변경을 포함할 때만 이 섹션을 포함합니다.

- 먼저 가능한 root cause를 식별합니다.
- 실용적이면 regression test를 추가하거나 갱신합니다.
- 증상만 덮는 패치를 피합니다.
- 수정이 root cause를 어떻게 해결하는지 설명합니다.

### Codex 설정과 AGENTS.md의 경계

`AGENTS.md`는 작업 맥락, 금지사항, 검증 기준, 절차를 주입합니다. 모델, reasoning effort, sandbox, approval policy 같은 런타임 설정은 일반적으로 `config.toml`에 둡니다.

- 사용자가 Codex 품질, 모델, reasoning 설정을 묻는 경우에만 설정 파일을 제안하거나 확인합니다.
- 사용자가 대상 `AGENTS.md` 생성만 요청했다면 `.codex/config.toml`이나 `~/.codex/config.toml`을 임의로 만들거나 수정하지 않습니다.
- 기본적으로 대상 `AGENTS.md` 본문에 config 예시를 넣지 않습니다. 필요하면 최종 응답의 별도 제안으로 분리합니다.
- 설정을 제안할 때는 먼저 대상의 기존 설정과 사용자 요청을 확인합니다.

### CLAUDE.md 링크

대상에 `CLAUDE.md`가 필요하다면 가능한 경우 `AGENTS.md`를 가리키는 심볼릭 링크로 생성합니다.

```sh
ln -s AGENTS.md CLAUDE.md
```

이미 `CLAUDE.md`가 존재한다면 먼저 종류를 확인합니다.

- 이미 `AGENTS.md`를 가리키는 심볼릭 링크라면 유지합니다.
- 일반 파일이거나 다른 대상을 가리키는 링크라면 임의로 덮어쓰지 말고 사용자에게 확인합니다.

Windows에서는 일반 사용자 권한으로 symlink 생성이 실패할 수 있습니다. 대상 환경에서 가능한 방법만 사용합니다.

1. 개발자 모드가 켜져 있거나 관리자 권한이 있다면 PowerShell에서 symlink를 만들 수 있습니다.

   ```powershell
   New-Item -ItemType SymbolicLink -Path CLAUDE.md -Target AGENTS.md
   ```

2. Git for Windows를 사용한다면 `git config core.symlinks`를 확인합니다.
3. symlink를 사용할 수 없다면 `CLAUDE.md`를 일반 파일 사본으로 둘 수 있습니다. 이 경우 대상 `AGENTS.md`나 contributing 문서에 두 파일을 함께 갱신해야 한다고 명시합니다.

작성 후 가능하면 아래를 확인합니다.

```sh
test -f AGENTS.md
git diff --check
git status --short
```

`CLAUDE.md`를 만들었다면 환경에 맞게 링크 또는 파일 동기화 상태도 확인합니다.

### 별도 문서 참조

대상에 세부 문서가 나뉘어 있다면 `AGENTS.md`에 모든 내용을 중복해서 옮기지 말고, 에이전트가 반드시 참고해야 할 문서와 읽는 순서를 명시합니다.

아래 유형의 문서가 있는지 확인합니다.

- 운영 적용 문서: 배포, 릴리스, 관리자 작업, 캐시 초기화, 운영 반영 후 확인 절차
- 런타임 또는 UI 계약 문서: DOM selector, API contract, schema, route, event, data attribute처럼 코드가 의존하는 외부 계약
- 스니펫 또는 플러그인 문서: 파일별 등록 위치, 실행 순서, hook priority, 필수 설정, 검증 명령
- 장애 처리 기록: 알림 채널, triage 원칙, 과거 판단, 상태 표시, 사용자의 승인 없이는 수행하면 안 되는 조치
- 문서 원본 관리: 생성 파일과 원본 파일, 수동 동기화가 필요한 파일, README 생성 규칙
- 성능 진단 문서: 재현 가능한 측정 방법, 산출물 위치, 해석 시 주의할 flag나 환경
- 보안 문서: secret 주입 위치, 공개 가능한 값과 비공개 값의 경계, 운영 접근 주의사항

별도 문서를 발견하면 대상 `AGENTS.md`에 다음처럼 요약합니다. 실제 파일이 없다면 예시 경로를 유지하지 않습니다.

```md
## 주요 참고 문서

- `docs/operations.md`: 배포와 운영 반영 절차를 확인합니다.
- `docs/contracts.md`: 코드가 의존하는 외부 마크업/API/schema 계약을 확인합니다.
```

세부 문서에 이미 있는 표나 긴 운영 기록은 `AGENTS.md`에 그대로 복제하지 않습니다. 대신 어떤 작업에서 어느 문서를 먼저 읽어야 하는지, 문서 갱신이 필요한 조건이 무엇인지 적습니다.

### 반복 실수 반영

같은 실수가 반복되면 규칙을 즉시 추가하지 말고 먼저 회고와 제안을 분리합니다.

- 잘못 판단한 부분을 한 문장으로 요약합니다.
- 같은 실수를 막을 구체적인 규칙을 제안합니다.
- 사용자 승인 없이 대상 `AGENTS.md`나 이 저장소의 `AGENTS.md`를 자동으로 업데이트하지 않습니다.
- 추상적인 원칙보다 다음 작업부터 적용할 수 있는 규칙을 우선합니다.

### 로드 확인

사용자가 `AGENTS.md`가 실제로 읽히는지 확인하라고 요청한 경우에만 대상 환경에서 검증 명령을 실행하거나 설명합니다. 사용 가능한 도구와 승인 정책에 맞는 명령만 사용하고, 실행하지 않은 결과를 성공으로 보고하지 않습니다.
