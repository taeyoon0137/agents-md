# agents-md 작업 지침

이 저장소에서는 아래 기준을 우선합니다. 사용자가 별도로 다른 지시를 내리면 최신 사용자 지시를 따릅니다.

## 기본 응답 원칙

- 항상 한국어 존댓말로 응답합니다.
- 결론을 먼저 말하고, 확인하지 못한 내용은 명확히 분리합니다.
- 불필요한 칭찬, 감탄, 장식적 표현은 사용하지 않습니다.
- 사용자가 커밋을 명시적으로 요청하지 않으면 커밋하지 않습니다.
- 이미 수정된 파일이 있으면 사용자 또는 이전 작업의 변경으로 간주하고 임의로 되돌리지 않습니다.

## 저장소 역할

`agents-md`는 개인 프로젝트에서 사용할 `AGENTS.md` 생성 지침을 관리하는 저장소입니다.

- 이 저장소의 `AGENTS.md`는 이 저장소 자체의 작업 지침이면서, 다른 저장소의 `AGENTS.md`를 생성하기 위한 프리셋 역할도 합니다.
- `CLAUDE.md`는 별도 파일로 복제하지 않고 `AGENTS.md`를 가리키는 심볼릭 링크로 유지합니다.

## 작업 시작 체크

작업 시작 시 다음을 먼저 확인합니다.

- `git status --short`
- 변경 대상 파일의 기존 문체와 구조
- 사용자의 목적이 이 저장소 유지보수인지, 다른 저장소의 `AGENTS.md` 생성인지
- `CLAUDE.md`가 `AGENTS.md`를 가리키는 심볼릭 링크인지
- 작업에 참고해야 할 별도 문서가 있는지

## 목적 분기

작업을 시작할 때 사용자의 목적을 먼저 분기합니다.

### 이 저장소 유지보수

사용자가 `agents-md` 자체의 문서, 지침, 구조, git 설정을 바꾸려는 경우입니다.

- 이 저장소의 source of truth는 `AGENTS.md`입니다.
- 별도 프리셋 파일을 만들지 않습니다.
- `CLAUDE.md`는 항상 `AGENTS.md`를 가리키는 심볼릭 링크로 유지합니다.
- 범용 개인 프로젝트 기준으로 작성하고, 특정 회사, 고객사, 비공개 제품명, 운영 채널, 내부 URL을 넣지 않습니다.
- 이 저장소의 문서가 다른 저장소 생성 지침과 충돌하지 않도록 유지합니다.
- 커밋 규칙, 문서 참조 규칙, 검증 규칙처럼 모든 대상 저장소에 유용한 항목은 이 파일에 유지합니다.

### 다른 저장소 AGENTS.md 생성

사용자가 다른 저장소의 `AGENTS.md`를 만들거나 갱신하려는 경우입니다. 이때 이 파일을 그대로 복사하지 말고, 대상 저장소를 조사한 뒤 실제 맥락에 맞는 `AGENTS.md`를 작성합니다.

대상 저장소에서 먼저 확인합니다.

- `git status --short`
- README, docs, package manifest, lockfile, 설정 파일
- 사용 언어와 프레임워크
- 빌드, 테스트, 린트, 포맷 명령어
- 배포나 운영 반영 방식
- 생성 파일과 source of truth의 관계
- 기존 `AGENTS.md`, `CLAUDE.md` 존재 여부
- 작업에 참고해야 할 별도 문서의 위치와 우선순위

대상 저장소의 `AGENTS.md`에는 아래 원칙을 반영합니다.

- 항상 한국어 존댓말로 응답하도록 지시합니다.
- 결론을 먼저 말하고, 확인하지 못한 내용은 명확히 분리하도록 지시합니다.
- 사용자가 커밋을 명시적으로 요청하지 않으면 커밋하지 않도록 지시합니다.
- 이미 수정된 파일은 사용자 또는 이전 작업의 변경으로 보고 임의로 되돌리지 않도록 지시합니다.
- 대상 저장소의 실제 파일 구조, 명령어, 검증 기준을 구체적으로 적습니다.
- secret, token, credential을 저장소에 기록하지 않도록 지시합니다.
- 요청과 직접 관련 없는 리팩터링을 하지 않도록 지시합니다.
- 확인하지 못한 명령어를 성공한 것처럼 보고하지 않도록 지시합니다.
- 커밋은 사용자가 명시적으로 요청한 경우에만 만들도록 지시합니다.

대상 저장소에 `CLAUDE.md`를 만들 때는 `AGENTS.md`를 가리키는 심볼릭 링크로 생성합니다.

```sh
ln -s AGENTS.md CLAUDE.md
```

이미 `CLAUDE.md`가 존재한다면 먼저 종류를 확인합니다.

- 이미 `AGENTS.md`를 가리키는 심볼릭 링크라면 유지합니다.
- 일반 파일이거나 다른 대상을 가리키는 링크라면 임의로 덮어쓰지 말고 사용자에게 확인합니다.

작성 후 대상 저장소에서 아래를 확인합니다.

```sh
test -f AGENTS.md
test -L CLAUDE.md
test "$(readlink CLAUDE.md)" = "AGENTS.md"
git diff --check
git status --short
```

## 대상 저장소 AGENTS.md 작성 기준

다른 저장소의 `AGENTS.md`를 만들 때는 아래 섹션을 기본 후보로 삼습니다. 대상 저장소에 해당하지 않는 항목은 제거하고, 해당하는 항목은 실제 파일명과 명령어로 구체화합니다.

- 기본 응답 원칙
- 저장소 역할
- 작업 시작 체크
- 커밋 원칙
- 문서와 source of truth
- 개발 기준
- 검증 기준
- 배포 또는 운영 반영 기준
- 장애, 알림, 모니터링, 사용자 문의 처리 기준
- 보안과 secret 관리

## 별도 문서 참조 기준

대상 저장소에 세부 문서가 나뉘어 있다면 `AGENTS.md`는 모든 내용을 중복해서 옮기지 말고, 에이전트가 반드시 참고해야 할 문서와 읽는 순서를 명시합니다.

아래 유형의 문서가 있는지 확인합니다.

- 운영 적용 문서: 배포, 릴리스, 관리자 작업, 캐시 초기화, 운영 반영 후 확인 절차
- 런타임 또는 UI 계약 문서: DOM selector, API contract, schema, route, event, data attribute처럼 코드가 의존하는 외부 계약
- 스니펫 또는 플러그인 문서: 파일별 등록 위치, 실행 순서, hook priority, 필수 설정, 검증 명령
- 장애 처리 기록: 알림 채널, triage 원칙, 과거 판단, 상태 표시, 사용자의 승인 없이는 수행하면 안 되는 조치
- 문서 원본 관리: 생성 파일과 원본 파일, 수동 동기화가 필요한 파일, README 생성 규칙
- 성능 진단 문서: 재현 가능한 측정 방법, 산출물 위치, 해석 시 주의할 flag나 환경
- 보안 문서: secret 주입 위치, 공개 가능한 값과 비공개 값의 경계, 운영 접근 주의사항

별도 문서를 발견하면 대상 저장소의 `AGENTS.md`에 다음처럼 요약합니다.

```md
## 주요 참고 문서

- `docs/operations.md`: 배포와 운영 반영 절차를 확인합니다.
- `docs/contracts.md`: 코드가 의존하는 외부 마크업/API/schema 계약을 확인합니다.
- `docs/snippets.md`: 운영 스니펫의 등록 위치, 우선순위, 필수 설정을 확인합니다.
- `docs/incident-notes.md`: 알림이나 장애 항목을 분류하기 전 기존 처리 이력을 확인합니다.
```

세부 문서에 이미 있는 표나 긴 운영 기록은 `AGENTS.md`에 그대로 복제하지 않습니다. 대신 어떤 작업에서 어느 문서를 먼저 읽어야 하는지, 문서 갱신이 필요한 조건이 무엇인지 적습니다.

## 문서 작성 기준

- 문서는 한국어를 기본으로 작성합니다.
- 절 제목은 짧고 명확하게 둡니다.
- 저장소별로 달라지는 내용과 범용 원칙을 섞지 않습니다.
- 확인하지 못한 명령어를 성공한 것처럼 쓰지 않습니다.
- 생성 파일은 직접 수정으로 끝내지 않고 원본 파일을 수정한 뒤 생성 명령을 실행하도록 지시합니다.
- 문서 원본과 코드 내 embedded text가 수동 동기화 관계라면 둘 다 갱신하도록 지시합니다.

## 커밋 원칙

커밋은 사용자가 요청한 경우에만 합니다. 다른 저장소의 `AGENTS.md`를 생성할 때도 같은 원칙을 넣습니다.

커밋을 만들 때는 역할과 목적별로 나눕니다.

- 기능 추가: `Feat: ...`
- 문서 갱신: `Docs: ...`
- 이름/구조 정리: `Refactor: ...`
- 버그 수정: `Fix: ...`
- 설정 변경: `Chore: ...`
- 테스트 추가/수정: `Test: ...`

최근 커밋이 있는 저장소에서는 기존 커밋 제목의 언어, 대소문자, prefix 스타일을 우선합니다. 기존 스타일을 확인할 수 없으면 위 형식을 기본값으로 사용합니다.

한 커밋에는 하나의 목적만 담습니다. 예를 들어 아래는 분리합니다.

- 기능 코드 변경
- 문서 생성본 갱신
- 설정 파일 변경
- spelling dictionary 또는 lockfile 갱신
- 운영 대응 기록 갱신

사용자가 커밋을 요청했더라도 unrelated change는 함께 stage하지 않습니다. 같은 파일에 사용자 변경과 작업 변경이 섞여 있으면 diff를 확인하고 필요한 hunks만 선별합니다.

## 검증 기준

변경 후 가능한 범위에서 아래를 실행합니다.

- `git diff --check`
- `git status --short`
- 심볼릭 링크 확인이 필요하면 `ls -l CLAUDE.md`

다른 저장소의 `AGENTS.md`를 생성할 때는 대상 저장소의 실제 명령어를 기준으로 검증 섹션을 작성합니다.

- README나 문서 생성본 변경: 해당 생성 명령
- TypeScript/JavaScript 변경: type check, lint, format check, test, build
- Python 변경: formatter, linter, type checker, test
- PHP 변경: `php -l <file>` if available
- 프런트엔드 변경: build 후 필요한 경우 브라우저에서 실제 화면 확인
- 배포/운영 관련 변경: 반영 전후 확인 절차와 rollback 또는 관측 기준

검증할 수 없는 항목은 성공처럼 말하지 말고 이유를 보고합니다.
